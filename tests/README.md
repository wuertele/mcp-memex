# Tests

Sprint 000 establishes a disposable local test platform for `mcp-memex`.
The supported entrypoint is:

```bash
./tests/run-tests.sh
```

The runner uses Docker Compose project name `memex-test`, starts a fresh
PostgreSQL + pgvector container and the mock inference service, waits for
readiness, runs the Deno smoke suite, and tears everything down with
`down -v --remove-orphans`.

## Prerequisites

- macOS container runtime:
  - Recommended: `brew install colima docker`
  - Alternative: Docker Desktop
- Deno:
  - `brew install deno`
- Optional PostgreSQL client tools for manual inspection:
  - `brew install postgresql@16`

No host PostgreSQL server or host `psql` binary is required for
`./tests/run-tests.sh`.

## Starting Colima

Start Colima before running the test harness:

```bash
colima start
```

Default Colima settings should be enough. If your machine is underprovisioned
or image builds feel slow, this also works:

```bash
colima start --cpu 4 --memory 4
```

The runner uses `docker info` as the canonical health check for the container
runtime. If `docker info` fails, start Colima or open Docker Desktop first.

## Running Tests

Run the full local harness from the repository root:

```bash
./tests/run-tests.sh
```

The runner is intentionally serial in Sprint 000. There is no
`--no-teardown` flag and no parallel mode. If you need to debug a live stack,
bring it up directly with Compose instead of modifying the runner.

## Fixed Ports

- PostgreSQL: `127.0.0.1:55432`
- Mock inference: `127.0.0.1:58000`

If either port is already bound, stop the conflicting process rather than
changing the repository defaults. The runner fails fast and points at:

```bash
lsof -i :55432
lsof -i :58000
```

## Manual Inspection

Bring up the stack without the runner when you want to inspect it manually:

```bash
docker compose -p memex-test -f tests/compose.yaml up -d --build --wait
docker compose -p memex-test -f tests/compose.yaml ps
curl -sS http://127.0.0.1:58000/health
docker compose -p memex-test -f tests/compose.yaml exec -T postgres \
  pg_isready -U memex_test -d memex_test
```

If you have `psql` installed, this connects to the disposable database:

```bash
psql postgresql://memex_test:memex_test@127.0.0.1:55432/memex_test
```

Tear the stack down explicitly when finished:

```bash
docker compose -p memex-test -f tests/compose.yaml down -v --remove-orphans
```

## Fixture Maintenance

### Canonicalization Fixtures

`tests/fixtures/canonicalization-cases.json` is the shared corpus for every
canonicalization implementation in this project. Each entry must contain:

- `name`
- `rule`
- `input`
- `expected`

Keep rule coverage for:

- `bom-stripping`
- `crlf-to-lf`
- `trailing-newline-collapse`
- `nfc`
- `internal-whitespace`
- `boundary`

Boundary cases for the empty string, a single newline, and whitespace-only
content must remain present. The file stays pure ASCII JSON with Unicode
escaped as `\uXXXX`.

NUL bytes are intentionally out of scope. PostgreSQL `text` columns cannot
store `\u0000`, so those fixtures would not round-trip through the SQL trigger
tests that consume this corpus in Sprint 001.

### Chat Fixtures

`tests/mock-inference/fixtures/chat.json` stores request/response pairs. At
service startup, each fixture request is canonicalized by recursively sorting
object keys, then hashed with SHA-256. Incoming chat requests are hashed the
same way, so key order in the fixture file does not affect lookup.

To add a new chat fixture:

1. Run the mock service.
2. Send the missing request to `POST /chat/completions`.
3. Copy the returned `error.request_hash` into your notes so you know which
   request was missing.
4. Add a new `{name, request, response}` entry to `chat.json`.
5. Restart the mock service and replay the request.

### Embedding Golden Fixtures

`tests/mock-inference/fixtures/embeddings.json` is a golden replay file. The
responses must be regenerated by calling the committed mock service after any
change to the embedding algorithm. Never hand-author or edit vector contents.

The expected workflow is:

1. Bring up the committed mock service.
2. `curl` the chosen `/embeddings` requests.
3. Save the exact JSON responses into `embeddings.json`.
4. Re-run the smoke suite and confirm replay passes.

Include at least one string-array request so the `input: string[]` path stays
covered.

## Colima Troubleshooting

- `docker info` is the first thing to check. If it fails, the runtime is not
  usable yet.
- If Colima behaves oddly, confirm the active Docker socket and context before
  debugging the repository.
- Multi-arch image selection is automatic. `pgvector/pgvector:pg16` and the
  pinned Deno image rely on published manifests rather than repo-specific
  architecture flags.

## Known Quirks

- `tests/compose.yaml` intentionally omits the Compose `version:` field because
  Compose v2 uses the latest spec automatically.
- The test harness is serial by design in Sprint 000.
- Use direct `docker compose ... up -d` for debugging instead of expecting a
  `--no-teardown` runner option.
- Git remote fixtures are not part of Sprint 000. They arrive with the sync
  daemon work later in the roadmap.
- The CI workflow is secret-free by design, so it runs on both `push` and
  `pull_request`, including forks.
